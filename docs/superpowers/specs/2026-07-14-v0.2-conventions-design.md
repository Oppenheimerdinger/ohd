## 1. `way-of-working` 스킬

용도: 이 하네스의 작업 규율 총람 — 세션이 "어떤 도구를 언제" 판단할 때의 라우터.
발화: 명시 요청("작업 방식", "way of working") + 새 프로젝트/온보딩 맥락.
description ~50단어.

### 품질 라우팅 표 (박제)

| 상황 | 도구 |
|---|---|
| 답이 아직 없음 (미해결 문제) | deep-solve (isolated/grounded 내부 라우팅) |
| 산출물이 이미 있음 | review-to-convergence |
| 산출물이 코드 diff | review-to-convergence가 **/code-review를 계기로** |
| 창의적/구조적 작업 완료 직후 | workflow 리뷰 (아래 기준) |

**workflow 리뷰 트리거 (기본값)**: 새 설계/새 모듈/알고리즘 선택처럼 "정답이
아니라 판단"이 들어간 작업이 (a) 파일 3개+ 또는 신규 로직 100줄+ 이거나 (b)
이후 작업이 그 위에 쌓이는 기반일 때 → multi-agent workflow 리뷰 권장. 미만이면
review-to-convergence 단일 리뷰어로 충분. (기준은 기본값 — 사용자가 조정)

### 두 force-multiplier (coordinator에서 이식, 기본 동작으로)

1. **위임 brief 규율**: 분리 가능 + 정답 존재 하위문제는 인라인으로 갈지 말고
   self-contained brief로 fresh agent에 위임. **brief는 검증된 정보만** —
   미검증이면 "unverified:"로 명시, 전제 반증을 초대. (가벼운 경우의 수동 위임;
   전체 루프가 필요하면 deep-solve로 승격)
2. **수렴 리뷰**: 실질 산출물은 독립 리뷰어 zero-finding까지 (review-to-convergence).
   합성 패턴: 위임으로 답 생산 → 독립 비평으로 검증 후 채택.

### 지속-루프 라우팅 (가벼운 것 먼저)

| 필요 | 추천 순서 |
|---|---|
| 주기 반복/폴링 | 내장 `/loop` (설치 0) |
| 예약 실행 | 내장 `schedule` |
| "될 때까지" 인세션 지속 | 그때만 OMC ralph — **설치가 무겁고 오발동 이력**을 명시 |

ralph 사용 시 필수 동반 지식: ① 키워드 훅 오발동 — 메시지에 "ralph"가 *언급*만
돼도 상태가 등록될 수 있음; 설계 논의/상태 질문은 ralph 대상이 아님을 판단하고
cancel ② cancel 절차: `/oh-my-claudecode:cancel` → 실패 시 `--force` → 그래도
안 잡히면 **CWD의 repo-로컬 `.omc/state/sessions/<id>/`를 직접 확인** (MCP state
툴이 다른 루트를 해석할 수 있음 — 2026-07-14 실사례 3회) 후 cancel 스킬의 bash
fallback ③ 자율 루프에는 반드시 종료 조건과 상한.

### 협업/오케스트레이션 규율

- **멀티노드/GitHub-truth**: 코드는 git으로만 이동 (commit→push→pull), scp 금지
  (스크래치 throwaway 제외), 노드 간 파일시스템 비공유 전제 — 다른 머신/에이전트에
  파일 경로를 건넬 땐 **내용을 같은 메시지에 동봉**.
- **희소자원 1레인**: 하나뿐인 자원(GPU 박스 등)은 한 레인으로 직렬화하고
  나머지를 병렬화; 그 자원 위의 작업은 bounded(timeout)+tracked+cleaned.
- **idle ≠ 죽음**: 조용한 팀메이트는 보통 작업 중 — 죽었다고 선언하거나 작업을
  죽이기 전에 긍정적 확인(git 활동, 수신 메시지) 필수.
- **superpowers 경유**: 창작은 brainstorming부터, 계획은 writing-plans, 실행은
  subagent-driven, 완료 주장 전 verification-before-completion. (superpowers가
  이 시점부터 실질 필수 — ohd-setup 라벨과 lock-step 정합 확인)

## 2. `campaign-land` 스킬 (일반화 이식)

umbrella-proj 원본의 7-phase land 의식에서 GPU/DFT/포크-핀 특수를 제거:

- **Phase 0 전제조건**: validation gate 녹색(가리킬 수 있는 증거), 부분 검증이면
  검증된 부분만 land + 잔여는 문서화된 후속으로 명시. 불확실하면 land 계획을
  사용자에게 제시하고 go를 받음.
- **Phase 1 작업트리 안전**: 전 브랜치 committed AND pushed 확인 (worktree별
  `git status` clean + `log origin/<br>..HEAD` empty). **reset --hard 전에
  앵커/문서 커밋+push** (단일 사본 worktree는 unpushed = reset 한 번에 소멸).
- **Phase 2 겹침 인지 재검증**: 검증은 자기 base 기준이지 병합 후 트렁크 기준이
  아님 — base 이후 land된 것과의 겹침 측정: **비겹침 → 재검증 없이 임의 순서 병합;
  겹침 → 병합 후 트렁크에서 충돌 집합만 재검증** (블랭킷 전체 재검증 아님).
- **Phase 3 diff 품질 게이트**: /code-review 실행 → 실 findings 수정 →
  **수정 후 재확인 없이 "고쳤다" 주장 금지**.
- **Phase 4 같은 land에서 문서 갱신**: living 문서는 같은 PR에서 갱신 안 하면
  조용히 드리프트. 상태문서에 최종 RESULT/VERDICT/후속 기입.
- **Phase 5 PR 병합 함정들** (원본 그대로 — 전부 범용): ⚠ 스택 PR base 확인
  (`gh pr view --json baseRefName`; base≠trunk면 MERGED 표시돼도 트렁크 미도달 —
  재타깃 + head-고유 파일이 트렁크에 있는지 확인) ⚠ `gh pr merge`는 서버사이드 →
  로컬 트렁크 stale → `fetch + reset --hard origin/<trunk>` ⚠ 순차 병합은 living
  문서에서 충돌 → **UNION으로 해소** (trunk를 브랜치로 merge, 양쪽 행 모두 보존,
  force-push 금지). 병합은 준비되는 대로 즉시, 배칭 금지.
- **Phase 6 교훈 증류 + 위생**: 비자명한 것만 메모리에; **병합 상태를 bare
  fact로 쓰지 말 것** — "as of <날짜> pushed NOT merged — 재유도하라"로 쓰고
  병합과 같은 패스에서 MERGED로 갱신 (팬텀 백로그 방지). land 시 claude-md-sanity
  실행.
- **Phase 7 정리**: 전 PR 병합 후에만 clean (worktree/브랜치 삭제 — 스폰한 자가
  teardown 소유). ⚠ 동일 argv 병렬 프로세스는 `pkill -f` 금지, PID로.
- **다중 캠페인**: 겹침으로 그룹핑 — 비겹침 임의 순서, 겹침 스택은 base부터
  순차 + 각 단계 재검증.
- **종속 repo pin (옵션 절)**: 포크/종속 repo가 있는 프로젝트만 — 종속 PR 먼저 →
  pin 갱신 → 본체 PR (campaign.sh의 pin 서브커맨드가 있으면 사용).
- **"이 스킬이 하지 않는 것"** 절 유지: 정답성 판단(validation gate 몫), 미확인
  결과 land, 의심스러울 땐 계획 제시 후 go.
- CLI 대응: **프로젝트에 campaign.sh(또는 동급)가 있으면 사용, 없으면 위
  phase들의 원시 git/gh 커맨드로** — 스킬이 양쪽 경로 다 제시.

## 3. `campaign-status` 스킬 (거의 그대로 이식)

- 존재 이유: **병합 상태는 git 그래프 + PR API의 속성이지 메모의 속성이 아님** —
  매번 재유도. 메모/문서/기억으로 "merged?"에 답하지 말 것.
- **squash-merge 함정**: GitHub 기본 병합은 새 커밋 생성 → 브랜치가 트렁크의
  조상이 아님 → `merge-base --is-ancestor`만 보면 진짜 병합된 브랜치가 UNMERGED
  → 팬텀 백로그.
- 판정: **MERGED = ancestry MERGED OR MERGED PR 존재**; UNMERGED는 둘 다 부정 +
  브랜치 실존일 때만.
- 재조정 스크립트 (BR/REPO/TRUNK 파라미터): fetch --prune → step 0 브랜치 실존
  먼저 → step 1 ancestry → step 2 `gh pr list --head $BR --state all --json
  number,state,mergedAt,baseRefName` → step 3 (신호 불일치 또는 base≠trunk 시)
  head-고유 토큰의 트렁크 도달 증명.
- 판정표 5종 (MERGED-ancestry / MERGED-squash / STACKED / UNMERGED / NO-BRANCH)
  + "CLOSED에 mergedAt=null = 미병합" + 다중 PR이면 하나라도 MERGED면 MERGED.
- 실행 시점: land 백로그 주장 전 / pin·clean 전 / 재실행·리베이스 전 / 어떤
  노트가 병합 상태를 단언할 때마다.

## 4. campaign.sh 템플릿 (assets/campaign.sh) + 드롭인 가이드

- **인스턴스화 방식 확정: config 헤더** (sed 템플릿팅 아님) — 파일 상단 변수
  블록, env로 오버라이드 가능 (umbrella-proj 원본과 동일 패턴):
  `TRUNK` (기본 main) / `WT_ROOT` (기본 $HOME/wt) / `NAMING` (free|numbered,
  기본 free) / `MERGE_MODEL` (coordinator|review-gate, 기본: 질문) /
  `STATE_DOC_DIR` (기본 docs/campaigns) / `MERGE_MODEL` 기본 coordinator
  (드롭인 인터뷰가 반드시 질문으로 확인) / `DEP_REPO`+`DEP_TRUNK` (기본 없음 —
  있으면 pin 서브커맨드 활성).
- 서브커맨드: `new | land | clean | abort | status | list` (+ DEP 있으면 `pin`).
  clean은 병합 확인(`--is-ancestor` + squash 대비 PR API 안내) 후에만; abort는
  원격 브랜치 보존. status/list는 campaign-status 스킬의 스크립트를 내장.
- **드롭인 가이드 (docs/campaign-dropin.md)**: 질문 기반 — 각 파라미터를
  질문+추천 기본값으로 묻고 config 헤더를 채움 → `tools/campaign.sh`로 복사 →
  트렁크 보호 훅(`tools/install-hooks.sh` 동봉, docs/**·*.md만 허용) 설치 →
  스모크 (new→abort 왕복). way-of-working가 이 가이드를 가리킴.
- **셀프 호스팅**: 드롭인 가이드를 이 repo에 적용하되 **트렁크 보호 훅은 생략**
  (이 repo는 v0.1 spec부터 의도적 main 직행 개발 — 훅을 깔면 플러그인 개발
  자체가 막힘; 가이드에 "훅 생략" 선택지가 있음을 이 케이스가 증명). 채택 범위 =
  campaign.sh 라이프사이클 + 상태문서. MERGE_MODEL=coordinator. 이 적용 과정
  자체가 가이드의 첫 실전 검증.
